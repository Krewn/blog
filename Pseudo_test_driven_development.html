Pseudo test driven development:

Let me start my saying that I may be using a slightly loser definition of test, than a lot of people might be used to. Traditionally tests assert some truthiness to indicate if they have passed or failed, and when we test we run all the tests. Personally I find that this requirement can make test writing and using tests cumbersome, especially when you are writing front end features, where passing means "looks good" and failing means "looks bad", and executing a test can fill up a display window.

So what do we lose by testing without asserting? For those of you familiar with tests the answer seems obvious, if your tests don't assert the developer must manually inspect at run-time to ensure that the results match the expected for each test. Whereas if tests have truthiness these assertions can be aggregated and interrogated directly without the developer checking each test off of a list. Unfortunately though these sort of tests don't  serve too well as sample code for using the methods that you have just developed, being all filled up by the assertion part!

That takes us to what you gain by testing without assertion, and my personal workflow is as follows:

    *Decide What I'm Doing -> Code -> WriteTest -> runTest <-> Debug ->*

In this way tests are not built to spec but rather a built to demo newly built code. While it’s been said that “The only time that ‘implementation’ comes before ‘test’ is in the dictionary.” writing post facto tests makes the purpose of your code more obvious because a test demonstrates a use-case and implementation without any additional anything. Therefor by reading these sort of tests you learn how to use the code. Secondly I will often neglect to run all my tests every time, instead opting to focus on one at a time, unless I edit code involved in a previous test. This keeps me focused and while the threat of bugs popping up mysteriously in old code is real, when I pay attention to where I am writing I find it doesn't often happen.

So What does the classic test driven process look like and how does it differ? Let's take a look:

    *WriteTests -> Code -> runTests <-> Dubug -> *

Having the test first makes it more of a boundary and less of a frame for the code devs write.  Notice the core of the process `Code -> runTest <-> Dubug -> *` remains the same.  This is because, in my opinion, test cases are the only way to actually demonstrate and ensure that your code works.

And at the end of the day, what else matters?
