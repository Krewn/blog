Basics of Web Development

“JavaScript, not to be confused with Java, was created in 10 days in May 1995 by Brendan Eich.”(A Short History of JavaScript)  In terms of functionality this provided a great step forward in web applications ”Vannevar Bush first proposed the basics of hypertext in 1945” (The History of html) “1989: Tim Berners-Lee invents the Web with HTML as its publishing language” (A short History of Html).

From it’s origin the web has been about providing users a good interface to interact with a remote resource. At its essence the DOCUMENTS represent whole sets of information arranged from a set of packets. The document model in modern web development involves an environment of input output devices interacting with a document presented as the user interface.

Applications are a dynamic model. Programmers designe sequences of interactions between applications, hence the term Application Program Interfaces (API) for the practice of providing methods to interact with a standing product.

Each web application is comprised of a code base which may include references or imports from external sources as well as programmatic interactions through API calls which are generally composed of an number of  Web Service Protocols .

At any given time the state of the user interface can be at a glance summarized by a static wire frame. After using a web site for any amount of time a seasoned web developer could easily reproduce the underlying structure of HTML laying out the elements of a page, and the visual details with CSS. These two components, the model and the view are only a piece of the puzzle however. To make a website tick you also need JavaScript or JS. JS is the controller, the third piece in the trifecta. JS can manipulate the HTML in order to provide dynamic experiences in the browser.

When a program interactions with another application in the browser it is JS that pulls the trigger. Sending receiving and manipulating the user interface in response is all JS. However it is important here to acknowledge the boundary of what we are doing. While the document can be manipulated by the JS is is important to acknowledge that the browser can be created with practically any language, and those programs such as Chrome, MozillaFireFox, Safari, and Microsoft Edge, are not limited the model view controller paradigm, they execute and interpret web protocols with code written in low level languages such as C. Furthermore a variety of frameworks exist to serve and manipulate documents, one category of these frameworks is the JS family which includes meteor, angular, and other JS libraries such as Jquery.  Another family of frameworks is rooted in php but today we will focus on the Model View Controller with HTML CSS and JS.

So what does HTML look like? Well because we live in the future our editors will color code our code for us and there are several whitespace conventions regarding the use of newlines and tabs this makes it easier to understand what you are looking at while reading code.  Additionally the browser provides a developer console that allows users to view the code behind any page. You can open the developer console by hitting ctrl+shift+i or by right-clicking and selecting `inspect element` from the browser menu.
Now let’s get down to the nomenclature of the HTML document. We have had our first glance at CSS style, an HTML table and JS from a distance. But let’s look at a simpler Html File First.

To make things as obvious as possible I drew in a few lines to connect the opening and closing tags of 3 of the elements in this simple html document. Html documents are composed of tags. With the exception of the first line <!DOCTYPE html> each element in the document has an opening and closing tag. Looking a little closer at the first example we see that in addition to innerhtml, beginning and ending tags  elements may also have some attributes. For example the <td> or table divisions had their id attribute set. You may also notice that within the style and script tags the contain innerhtml that differs from the other tag. This is because the innerhtml for a style tag is interpreted as  as a cascading style sheet (CSS) and the inner html of a script tag is interpreted as javascript (JS). It is also possible to use style and script tags with a src (source) attribute, this way we can reference centralized versions of commonly used scripts like Jquery.  

<script src="https://ajax.googleapis.com/ajax/libs/jquery/1.12.4/jquery.min.js"></script>

So at this point there should be no ambiguity about what an element(<>...</>), attribute(a=b), style(<style>css</style>), script(<script>JS</script>), and tag(<>) are.
